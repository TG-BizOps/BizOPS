왜 별도 AI 챗봇이 아닌가

최근 소비자 조사에서 일관되게 나타나는 트렌드: 고객의 상당수가 "AI 봇과 대화하고 있다"는 것을 인지하는 순간 신뢰도가 떨어지고, 처음부터 사람 상담원을 요구하는 비율이 높아집니다. 특히 결제/환불 같은 민감한 이슈에서는 "봇이 내 돈 문제를 제대로 처리할 수 있을까"라는 불안감이 큽니다. Toomics는 10개 마켓, 다국어 고객을 상대하기 때문에 이 거부감이 더 크게 작용합니다. 유럽 고객(DE, FR)은 특히 개인정보와 자동화 처리에 민감합니다.

비교 항목 별도 AI 챗봇 (프로토타입) Zendesk 투명 통합 (현재 방향)
고객 경험 새로운 채팅 UI 적응 필요, "봇과 대화 중" 인지 기존 Zendesk 이메일/폼 그대로, AI 인지 불가
고객 신뢰도 "AI가 대답하네" → 거부감, 사람 요청 증가 변화 없음, 그냥 답변이 빨라졌을 뿐
고객 이탈 리스크 봇 답변 불만족 → "사람 연결해줘" 반복 → 이중 처리 없음 — 기존 CS 채널과 동일
상담원 워크플로우 Zendesk + 챗봇 이중 관리 대시보드 하나로 통합
학습 비용 새 시스템 교육 필요 기존 Zendesk 워크플로우 + 대시보드만 추가
도입 리스크 높음 — 고객 대면 변경 낮음 — 백엔드만 변경, 프론트 그대로
점진적 확대 어려움 — 챗봇 ON/OFF 이분법 쉬움 — L3부터 자동화, 카테고리별 확대
품질 통제 봇이 바로 고객에게 노출 상담원 확인 단계 포함 (Phase 1)
GDPR/개인정보 새 채널 = 새 데이터 처리 동의 필요 가능 기존 Zendesk 데이터 처리 동의 범위 내

현재 방향의 핵심 장점
👤
1. 고객 거부감 제로
고객은 AI와 대화하고 있다는 사실을 모릅니다. 기존에 하던 대로 Zendesk으로 문의하고, 기존에 받던 대로 이메일로 답변 받습니다. "AI 봇입니다, 무엇을 도와드릴까요?" 같은 인위적인 경험이 없습니다.
🛡️
2. 운영 리스크 최소화
Phase 1에서 상담원이 모든 AI Draft를 확인하므로, 잘못된 답변이 고객에게 나갈 위험이 없습니다. 충분히 검증된 카테고리(L3)부터 점진적으로 자동화를 확대하는 구조입니다.
⚡
3. 상담원 업무 부담 실질적 감소
별도 챗봇은 "봇이 못 풀면 결국 사람이 처리"라서 오히려 업무가 늘 수 있습니다. 현재 방향은 AI가 초안을 만들어주니 상담원은 확인 + 클릭만 하면 됩니다. L3 자동화 후에는 복잡한 건에만 집중할 수 있습니다.
📊
4. 데이터 자연 축적
매일 처리되는 실제 티켓이 곧 AI 학습 데이터입니다. 별도 챗봇은 사용자가 안 쓰면 데이터가 안 쌓이지만, 이 구조는 기존 CS 운영이 곧 AI 고도화 데이터입니다.

Phase 2 전환 조건 및 기술 요구사항
📊
전환 조건 1: AI Draft 품질 검증
Phase 1에서 충분한 양의 AI Draft가 생성되고, 상담원 수정률이 일정 수준 이하로 떨어져야 합니다. L3 카테고리에서 상담원이 수정 없이 발송하는 비율이 90%+ 달성 시 전환 가능합니다.
수정률 <10% L3 무수정 발송 90%+
🏷️
전환 조건 2: L3 카테고리 정의
자동 발송 대상이 될 L3 카테고리를 명확히 정의해야 합니다. 구독 해지 확인, 비밀번호 리셋 안내, FAQ 안내 등 정형화된 답변만 해당됩니다.
카테고리 정의서 답변 템플릿 검증
🔌
기술 요구사항 1: Zendesk API 자동 발송
현재 대시보드에서 상담원이 클릭해서 발송하는 구조를, 조건 충족 시 자동으로 Zendesk API를 호출하여 답변을 발송하는 워커(worker) 또는 웹훅(webhook) 기반으로 확장해야 합니다.
Zendesk Trigger/Webhook Background Worker
🔒
기술 요구사항 2: 품질 게이트
자동 발송 전 품질 점수가 임계값(95%) 미만이면 반드시 상담원 큐로 라우팅하는 안전장치가 필요합니다. 스팸 감지, 운영자 확인 트리거 발동 시에도 자동 발송을 차단합니다.
Quality Gate 95% Spam/Verification Block
📈
기술 요구사항 3: 모니터링 및 롤백
자동 발송 후 고객 재문의율, CSAT 점수, 에스컬레이션율을 실시간 모니터링하고, 이상 징후 발견 시 특정 카테고리의 자동화를 즉시 해제하는 롤백 메커니즘이 필요합니다.
재문의율 모니터링 카테고리별 롤백
